\section{Selection of Planners}
\label{sec:setup:planners}

To test how different planning architectures respond to structural changes in domain models, this study requires planners that solve \gls{pddl} problems in fundamentally different ways.
The four planners used in this study were selected from an initial pool of 35 classical planners through a strict four-step filtering process.

\subsection{Filtering Methodology}

\begin{description}[style=nextline, leftmargin=1.5cm]
  \item[\textbf{Filter~1} (Agile Planning, No Portfolios):]
    Because the primary focus is on agile planning---finding a valid plan as fast as possible, regardless of cost---planners designed for the optimal track were excluded.
    Following the methodology of \citeauthor{Elis2025}~\cite{Elis2025}, portfolio planners were also excluded.
    Portfolio planners combine multiple algorithms and switch between them during execution; if an \gls{llm} modification improved the speed of such a planner, it would be impossible to determine which specific underlying architecture actually benefited.

  \item[\textbf{Filter~2} (Proven Performance):]
    Surviving planners were required to have a demonstrated track record: either winning recent \gls{ipc} Agile or Satisficing tracks, or having been extensively validated as strong baselines in prior automated planning research.

  \item[\textbf{Filter~3} (Reproducibility):]
    Planners were required to be open-source, deterministic (no reliance on random chance), and capable of running in isolated Docker containers~\cite{Merkel2014} to ensure fair and repeatable experiments.

  \item[\textbf{Filter~4} (Distinct Architectures):]
    This was the most critical filter.
    An analysis of the \gls{ipc}~2023 Agile leaderboard reveals that the vast majority of top-performing planners are variations of the same underlying engine (Fast Downward~\cite{Helmert2006}).
    Selecting the top four planners from such a list would effectively test the same architecture four times.
    To demonstrate differential architectural sensitivity, this study requires engines that process \gls{pddl} through fundamentally different pipelines.
    This filter also excludes hybrid planners (e.g., Scorpion Maidu) to ensure that pure architectures are tested.
\end{description}

\section{Selection of Domains}
\label{sec:setup:domains}

The benchmark domains used in this study were selected through a multi-stage filtering process designed to maximise structural diversity while satisfying the practical constraints of the experimental pipeline.

\subsection{Source Aggregation}

The initial candidate pool was assembled from multiple authoritative repositories spanning over 25 years of \gls{ipc} history (IPC~1998 through IPC~2023).
Sources include the official IPC archives, the AutoPlanBench benchmark suite, and the domain sets used by \citeauthor{Elis2025}~\cite{Elis2025} and \citeauthor{Georgievski2025}~\cite{Georgievski2025}.
This temporal breadth ensures that the dataset captures both foundational benchmarks---enabling direct comparison with prior work---and contemporary domains reflecting modern design practices.

\subsection{Filtering Criteria}

Candidate domains were subjected to three sequential filters:

\begin{description}[style=nextline, leftmargin=1.5cm]
  \item[\textbf{Filter~1} (Syntactic Compatibility):]
    Domains were required to parse successfully under all four target planner architectures.
    For example, DecStar lacks support for axioms (derived predicates), immediately disqualifying domains such as PSR-Small.
    Similarly, DecStar falls back to standard search if it encounters conditional effects, necessitating careful domain parsing checks to ensure the isolated architectural mechanics of each planner are genuinely evaluated.

  \item[\textbf{Filter~2} (Pipeline Compatibility):]
    The LLM-Modulo feedback loop (\Cref{sec:design:stage3}) requires a single, static \texttt{domain.pddl} file per domain.
    Domains using a ``domain-per-problem'' compilation model (e.g., Quantum Circuit Layout Synthesis, where problem-specific constants are hardcoded into individual domain files) or requiring mandatory external normalisers that overwrite \gls{llm} modifications (e.g., Rubik's Cube) were excluded to preserve the integrity of the structural feedback loop.

  \item[\textbf{Filter~3} (Structural Redundancy Reduction):]
    Among the surviving candidates, redundant overlaps were eliminated to maximise structural variance within a feasible scope.
    For instance, Blocksworld and Depots share significant block-stacking overlap; Gripper and Snake both occupy the untyped space; and Termes overlaps structurally with Snake in predicate count and maximum arity.
    In each case, the domain with the richer structural profile was retained.
\end{description}

\subsection{Structural Quantification}

To objectively verify that the selected domains offer a diverse range of structural challenges, we employ a domain-richness quantification framework adapted from HTN planning literature~\cite{Georgievski2026}.
Each domain is evaluated across five metrics:
(i)~$\lvert A \rvert$, the number of unique action operators;
(ii)~$\lvert P \rvert$, the total number of distinct predicates;
(iii)~$\lvert P_m \rvert$, the number of relational predicates (those defining relationships between two or more objects);
(iv)~$\lvert O_t \rvert$, the number of distinct leaf object types; and
(v)~max arity, the highest number of parameters any single action requires.

\subsection{Final Domain Selection and Justification}

The final five-domain subset retains the full range of every quantified metric, demonstrating that structural diversity was preserved despite the strict filtering process.
Each domain occupies a distinct niche in the structural spectrum:

\begin{description}[style=nextline, leftmargin=1.5cm]
  \item[VisitAll (IPC 2014) --- The Minimal Baseline:]
    A near-minimal domain with a single action, only three predicates, and a maximum arity of two, encoding a pure graph-reachability problem.
    Its minimal syntactic footprint ensures rapid, reliable telemetry generation for the LLM-Modulo loop without timing out.
    It provides a pure graph-reachability test where width-based search (\gls{bfws}) should strongly contrast with heuristic forward search (\gls{lama}).

  \item[Snake (IPC 2018) --- The Untyped, Negative-Precondition Challenge:]
    Selected over Termes and Gripper to represent the untyped domain space ($\lvert O_t \rvert = 0$).
    Snake introduces dead-end states and requires negative preconditions, creating severe structural stress that differentiates planners.
    Gripper was excluded because its trivially small search space causes all four planners to solve instances near-instantly, producing flat, undifferentiated telemetry.

  \item[Ricochet Robots (IPC 2023) --- The State-of-the-Art Mid-Range:]
    The sole representative of the latest \gls{ipc}, ensuring the dataset includes a contemporary benchmark.
    Its moderate type hierarchy ($\lvert O_t \rvert = 3$) fills the structural gap between the untyped domains and the deeply typed formulations.
    Furthermore, it is the only domain in the dataset that declares the \texttt{:action-costs} \gls{pddl} feature and utilises numeric fluent functions, broadening the syntactic expressivity of the final dataset.

  \item[Depots (IPC 2002) --- The Typed Logistics Mid-Range:]
    Selected over Blocksworld to provide a complex, three-level type hierarchy ($\lvert O_t \rvert = 6$) while maintaining a moderate predicate overhead.
    Its spatially decoupled nature (independent depot locations and distributors) serves as the ideal testbed for DecStar's decoupled search paradigm.

  \item[Barman (IPC 2014) --- The Maximal Complexity Anchor:]
    The absolute upper bound of grounding stress, with the highest action count ($\lvert A \rvert = 12$), deepest type hierarchy ($\lvert O_t \rvert = 7$), and highest action arity (6).
    Its dense, tightly coupled combinatorial space provides the strongest environment to test whether an \gls{llm} can isolate and optimise architecture-specific bottlenecks.
\end{description}

The final selection also achieves strong temporal diversity, spanning from IPC~2002 (Depots) through IPC~2014 (VisitAll, Barman) and IPC~2018 (Snake) to IPC~2023 (Ricochet Robots).

\section{Selection of Large Language Models}
\label{sec:setup:llms}

The four \glspl{llm} used in this study are partitioned into two architectural categories to prevent bias and ensure comprehensive evaluation across distinct computational paradigms.

\subsection{Categorisation Rationale}

Prior work by \citeauthor{Georgievski2025}~\cite{Georgievski2025} and \citeauthor{Elis2025}~\cite{Elis2025} demonstrates that the choice of the underlying \gls{llm} architecture is the single most statistically significant determinant of downstream planner efficiency, far outweighing the impact of varied prompting strategies or stochastic temperature fluctuations.
To capture this variance, the selected models are bifurcated into two categories:

\begin{description}[style=nextline, leftmargin=1.5cm]
  \item[Category~A --- Deep-Reasoning Architectures:]
    Models that allocate significant computational resources to inference-time reasoning, evaluating multiple logical pathways before output generation.
    Prior literature indicates that reasoning-optimised models consistently yield the most efficient structural reorderings for planner execution.

  \item[Category~B --- Coding Heavyweight Architectures:]
    Models designed for massive repository ingestion, syntax comprehension, and logical dependency tracking.
    Prior literature demonstrates that coding-heavy models achieve the highest rates of syntactic validity when reconstructing \gls{pddl}~\cite{Elis2025, Georgievski2025}.
\end{description}

\section{Selection of the General Prompt Strategy}
\label{sec:setup:general_prompt}

To establish a robust and reproducible general prompt baseline for Stage~1, the selection process draws directly on the empirical findings of \citeauthor{Georgievski2025}~\cite{Georgievski2025} and \citeauthor{Elis2025}~\cite{Elis2025}.

\subsection{Empirical Evidence from Prior Work}

The prior studies systematically evaluated five distinct prompting strategies across 700 generated domain configurations:
(i)~$\Pi_\text{ZS-S}$ (zero-shot-short): minimal instruction, no explicit rules;
(ii)~$\Pi_\text{ZS-L}$ (zero-shot-long): full domain-engineering rule set;
(iii)~$\Pi_\text{FS-S}$ (few-shot-short): rules plus one annotated example;
(iv)~$\Pi_\text{FS-L}$ (few-shot-long): rules plus two annotated examples; and
(v)~$\Pi_\text{CoT}$ (chain-of-thought): rules plus a multi-turn analysis-then-execution protocol.

The analysis revealed a counter-intuitive but significant finding: the simplest strategy, $\Pi_\text{ZS-S}$, decisively outperformed the more complex, heavily engineered prompts.
Specifically, $\Pi_\text{ZS-S}$ achieved:
\begin{itemize}
  \item A syntactic validity rate of 79.29\% (domains that parsed correctly without errors).
  \item A semantic equivalence validity rate of 55.00\% (domains that perfectly preserved the original semantics).
\end{itemize}

\noindent In contrast, providing the \gls{llm} with established domain-engineering rules ($\Pi_\text{ZS-L}$) decreased the full validity rate to 45.00\%, and providing annotated examples ($\Pi_\text{FS-S}$, $\Pi_\text{FS-L}$) resulted in full validity rates of only 46.43\% and 50.71\%, respectively.
The prior work concludes that if an \gls{llm} possesses the inherent capacity to understand \gls{pddl} semantics, minimal guidance prevents overfitting and reduces the likelihood of semantic drift.

\subsection{Adopted Strategy}

Given these empirical results, this study adopts the $\Pi_\text{ZS-S}$ (zero-shot-short) strategy as the Stage~1 baseline prompt, reproduced verbatim in Listing~\ref{lst:general_prompt}.
This choice ensures methodological continuity with the foundational studies and provides a strict, standardised benchmark against which the architecture-aware prompts (Stage~2) and feedback loop (Stage~3) can be measured.
The prompt encodes three constraints: (i)~strict semantic preservation; (ii)~raw output without explanatory text; and (iii)~general guidance referencing ``best practices from planning literature'' without specifying any particular planner.

\section{Selection of the Architecture-Aware Prompt Strategy}
\label{sec:setup:arch_aware_prompt}

Stage~2 extends the planner-agnostic approach of Stage~1 with architecture-aware prompting: four distinct, planner-tailored prompt variants that inject synthesised knowledge of each target planner's internal architecture into the \gls{llm}'s guidance component.

\subsection{The Research Gap}

The key finding from \citeauthor{Georgievski2025}~\cite{Georgievski2025} is that configuration effectiveness is highly planner-dependent.
Their experiments show that generic structural optimisation is insufficient---what helps one architecture can actively hurt another.
Their discussion explicitly calls for ``planner-specific feedback'' and notes that ``without access to planner internals, \glspl{llm} can only apply general domain configuration principles that happen to align with some planner architectures but conflict with others.''

This observation motivates a fundamental shift from planner-agnostic prompting to architecture-aware prompting.
Rather than relying on six generic domain-engineering rules derived from the literature~\cite{Vallati2021, Franco2019, FoxLong2003, Helmert2009, McCluskey2017}, the architecture-aware prompts encode rules that are traceable to specific internal components of each target planner.

\subsection{Prompt Template Extension}

The prior work defines a prompt template $\Pi = \langle r, C, D, G \rangle$, consisting of a role assignment~$r$, a constraint set~$C$, an input domain~$D$, and a guidance component~$G$.
This study extends the template to an architecture-aware variant $\Pi_\text{AA}^{P}$ for a specific planner~$P$:

\begin{equation}
  \Pi_\text{AA}^{P} = \langle r_\text{AA}, C, D, G_\text{AA}^{P} \rangle
\end{equation}

\noindent where $r_\text{AA}$ is an enhanced role assignment establishing the \gls{llm} as a \gls{pddl} expert with specific knowledge of planner~$P$'s architecture, $C$ is the unchanged constraint set from the prior work (semantic preservation), $D$ is the input domain, and $G_\text{AA}^{P}$ is the architecture-aware guidance component, consisting of:
\begin{itemize}
  \item $G_\text{arch}^{P}$: a synthesised description of planner~$P$'s internal pipeline and sensitivities.
  \item $G_\text{rules}^{P}$: architecture-derived reordering rules specific to planner~$P$.
\end{itemize}

\subsection{Design Principles}

The architecture-aware guidance $G_\text{AA}^{P}$ is constructed following three principles:

\begin{enumerate}
  \item \textbf{Traceability:} Every architecture-aware rule must be traceable to a specific internal component (parser, grounder, heuristic, search algorithm) of the target planner.
  \item \textbf{Differentiation:} Rules must differ meaningfully across planners---if a rule applies identically to all four planners, it is generic and should remain in the shared constraint set rather than in the architecture-specific guidance.
  \item \textbf{Actionability:} Rules must be expressed as concrete \gls{pddl} ordering directives that the \gls{llm} can mechanically apply, rather than abstract algorithmic descriptions.
\end{enumerate}

To eliminate the timing variance introduced by shared cloud infrastructure (noisy-neighbour effects and hypervisor steal time), all planner executions are performed on dedicated local hardware, as motivated in \Cref{sec:design:safeguards}.

\section{Hyperparameters and Reproducibility}
\label{sec:setup:hyperparameters}

All experimental parameters are centralised in a single YAML configuration file (\texttt{experiment\_config.yaml}) to ensure that the complete experimental setup can be reconstructed from a single source of truth.
This section documents the key hyperparameters and the reproducibility measures employed.

\subsection{LLM Hyperparameters}

\begin{description}[style=nextline, leftmargin=1.5cm]
  \item[\textbf{Temperature} ($T = 0.0$):]
    Deterministic greedy decoding is enforced for all models.
    This ensures that the same input always produces the same output, allowing performance differences to be attributed to the prompt design (general vs.\ architecture-aware) rather than to stochastic sampling variance.
    Both \citeauthor{Georgievski2025}~\cite{Georgievski2025} and \citeauthor{Elis2025}~\cite{Elis2025} found that temperature has no statistically significant effect on any performance metric (Friedman test, all $p > 0.2$, effect size $W < 0.07$), confirming that temperature variation adds computational cost without informational gain.
    An academic caveat applies for deep-reasoning models: while the temperature parameter is set to $0.0$, reasoning APIs (such as \texttt{deepseek-reasoner} and OpenAI's o-series) natively govern their own internal cognitive exploration parameters to evaluate multiple reasoning paths; in these cases, standardisation is ensured by utilising the model's default reasoning configuration.

  \item[\textbf{Top-$p$} (\texttt{top\_p} = 1.0):]
    At a temperature of $0.0$, the model always selects the most probable next token, making nucleus sampling functionally inactive.
    Setting \texttt{top\_p} = 1.0 serves as the default disabled value, ensuring that no probability mass truncation occurs before the greedy selection.

  \item[\textbf{Token generation limits} (\texttt{max\_completion\_tokens} = 8192):]
    The pipeline uses the modern \texttt{max\_completion\_tokens} parameter (rather than the deprecated \texttt{max\_tokens}) to dictate output boundaries across all provider APIs without triggering compatibility errors.

  \item[\textbf{Context memory} (Stateless):]
    Each \gls{llm} query is executed as an isolated REST API POST request.
    The model retains no conversational history or memory from previous domain formulations, guaranteeing zero cross-prompt data contamination during execution batches.
\end{description}

\subsection{Planner Execution Constraints}

Each Docker container is subject to the following resource constraints, following the \gls{ipc} Agile Track conventions:

\begin{itemize}
  \item \textbf{Time limit:} 360 seconds (6 minutes) per problem instance.
  \item \textbf{Memory limit:} 8~GB (with swap disabled via \texttt{--memory-swap=8g}).
  \item \textbf{CPU allocation:} 1 core per container (\texttt{--cpus=1.0}).
  \item \textbf{OOM handling:} The container is terminated immediately if the memory limit is exceeded (\texttt{oom-kill-disable=false}).
\end{itemize}

\subsection{Problem Instance Selection}

Each \gls{ipc} benchmark domain provides 20 problem instances.
To balance statistical power against computational cost, 15 out of the 20 instances are selected per domain using a fixed random seed (seed = 42), following the procedure described in the execution protocol.
The selected instances are:

\begin{center}
\texttt{[1, 2, 3, 4, 7, 8, 9, 11, 12, 13, 14, 16, 17, 18, 19]}
\end{center}

\noindent The same 15 instances are used across all five domains, all four planners, all four \glspl{llm}, and all experimental stages, ensuring that pairwise comparisons between stages are perfectly paired.
Only the selected instances are included in the repository's \texttt{instances/} directory, eliminating any ambiguity for future replication.

\subsection{API Version Pinning}

To ensure absolute longitudinal reproducibility, each \gls{llm} is accessed via its hardcoded API version string (see \Cref{tab:selected_llms}).
This means the model weights, tokeniser, and alignment layer are frozen for the duration of the experiment, even if the provider releases updates.
For providers that do not offer immutable snapshot endpoints (e.g., DeepSeek), reproducibility is enforced by documenting the exact timestamp of all API executions and monitoring the provider's update changelog.
